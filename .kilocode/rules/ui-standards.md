UI-СТАНДАРТЫ
Проект: MyIS — Единая информационная система подразделения

Версия: 2.0
Статус: Обязательный стандарт
Область действия: Вся система MyIS (Frontend, UX, AI-генерация UI)

Назначение документа

Настоящий документ определяет единые правила проектирования пользовательского интерфейса (UI) системы MyIS и является обязательным к применению при:

разработке новых экранов и модулей;

расширении и рефакторинге существующего интерфейса;

генерации UI с помощью ИИ;

проектировании интеграций, влияющих на пользовательский интерфейс.

Документ используется как:

UI-контракт между бизнесом, аналитикой и разработкой;

единый источник истины по структуре интерфейса;

контекст для ИИ при генерации UI (React + Ant Design);

основа для UX-ревью и приёмки.

1. Базовая концепция интерфейса MyIS
1.1 Object-Centric UI (ключевой принцип)

Интерфейс MyIS строится вокруг бизнес-объектов и их жизненного цикла, а не вокруг таблиц, форм или технических сущностей.

Примеры бизнес-объектов:

Заявка

Заказ

Номенклатурная позиция

Изделие

Партия

Несоответствие

Каждый бизнес-объект:

имеет уникальный идентификатор;

обладает жизненным циклом и статусом;

участвует в нескольких бизнес-доменных процессах;

имеет историю, документы, связи и действия.

UI обязан:

визуально отражать состояние объекта;

показывать доступные действия в зависимости от статуса и роли;

не требовать от пользователя знания структуры системы.

2. Роль-ориентированность интерфейса
2.1 UI как рабочее место роли

Интерфейс проектируется вокруг ролей и рабочих сценариев, а не вокруг структуры БД или модулей.

Пользователь должен:

видеть только релевантные его роли функции;

сразу понимать, что от него требуется;

иметь быстрый доступ к своим задачам и ответственности.

Запрещено:

универсальные экраны «для всех ролей»;

вывод всей информации «на всякий случай»;

интерфейс, требующий знания внутренних связей системы.

3. Единый App Shell (обязательный)
3.1 Назначение App Shell

App Shell — неизменяемый каркас интерфейса MyIS.
Все экраны и модули обязаны в него встраиваться.

3.2 Структура App Shell
3.2.1 Header (верхняя панель)

Глобальные элементы:

название системы;

текущий пользователь и активная роль;

уведомления и Inbox;

глобальный поиск;

пользовательские настройки и выход.

Запрещено:

размещать доменные функции;

размещать действия над объектами.

3.2.2 Left Navigation (левое меню)

Содержит только бизнес-домены верхнего уровня:

Requests

Customers

Procurement

Production

Warehouse

Engineering

Technology

Quality

References

Administration

Меню отражает бизнес-контуры, а не технические модули.

3.2.3 Content (рабочая область)

Единственное место, где:

отображаются журналы;

открываются карточки объектов;

выполняется работа пользователя.

4. Навигация по доменам
4.1 Домен = бизнес-контур

Каждый домен:

соответствует устойчивой зоне ответственности;

агрегирует связанные бизнес-объекты и процессы;

не дублирует функции других доменов.

Запрещено:

пункты меню вида «Сущности», «Таблицы», «Формы»;

дублирование одного объекта в разных доменах без смены контекста.

4.2 Вложенность меню

Допускается не более двух уровней вложенности.

Пример:

Requests
 ├─ Журнал заявок
 ├─ Мои заявки
 └─ На согласовании

5. Базовый паттерн экранов (обязательный)
5.1 Паттерн «Журнал → Карточка»

Любой бизнес-объект, имеющий жизненный цикл, обязан иметь:

Журнал (List / Journal) — обзор и навигация

Карточку (Details) — единое место работы с объектом

Дополнительные представления допускаются только после реализации этих двух.

5.2 Журнал (Journal)

Журнал — основной рабочий экран домена.

Обязательные элементы:

список объектов (таблица или карточки);

фильтры;

сортировка;

сохранённые представления;

Command Bar (верхняя панель действий).

Журнал не является формой редактирования.

Запрещено:

сложное редактирование данных прямо в журнале;

выполнение бизнес-действий без перехода в карточку.

5.3 Card-based Journal (допустимый вариант)

Для сущностей с выраженным жизненным циклом и роль-ориентированной работой
(заявки, задачи, согласования) допускается карточный журнал.

Карточный журнал:

каждая строка = карточка бизнес-объекта;

отображает статус, ключевые атрибуты и сроки;

не предназначен для редактирования;

всегда ведёт в карточку объекта.

6. Карточка объекта (Details)

Карточка — единое место истины по объекту.

Карточка всегда содержит:

заголовок: тип + номер/код + текущий статус;

Command Bar;

вкладки по стандарту.

7. Стандарт вкладок карточки (обязательный)
7.1 Базовый шаблон вкладок

Для всех ключевых объектов используется единый шаблон:

Общее
Основные атрибуты, статус, ключевые поля.

Состав / Связи
BOM, позиции, связи с заказами, заявками, партиями.

Документы
КД/ТД, файлы, версии, сканы.

История
История изменений, статусов, комментарии, аудит.

Задачи / Согласования
Назначенные действия, состояние исполнения.

Интеграции / Логи (по ролям)
Обмен с внешними системами, технические события.

Правило:
Если функциональность не укладывается ни в одну вкладку —
необходимо создавать новую сущность, а не новую вкладку.

8. Command Bar (обязательный)
8.1 Единое место действий

Все действия над объектом размещаются только в Command Bar:

изменение статуса;

отправка на согласование;

закрытие / отмена;

создание связанных объектов.

Запрещено:

дублировать действия в разных местах карточки;

размещать бизнес-действия внутри вкладок.

9. References, Administration и Settings
9.1 Разделение ответственности
Раздел	Назначение
User Settings	Личные настройки пользователя
References	Бизнес-справочники (MDM)
Administration	Система, роли, доступы
9.2 Справочники (References)

Справочники:

организуются иерархически;

используют паттерн «Журнал → Карточка»;

не содержат процессной логики.

10. Inbox и диспетчеризация

Каждый пользователь имеет единый Inbox, содержащий:

задачи согласования;

задачи исполнения;

просроченные действия;

возвраты и замечания.

Inbox — приоритетный экран после входа в систему.

11. Правила расширения интерфейса

При добавлении нового функционала порядок строго следующий:

Новое представление журнала

Новый фильтр или действие

Новая вкладка (только при необходимости)

Новый экран (только при новом сценарии)

Запрещено:

добавлять экраны без явного бизнес-сценария;

расширять UI «про запас».

12. Инструкция для ИИ (обязательная)

При генерации интерфейса ИИ обязан:

Использовать существующий App Shell.

Работать в рамках существующего домена.

Применять паттерн «Журнал → Карточка».

Использовать стандарт вкладок.

Размещать действия только в Command Bar.

Учитывать роли и статус объекта.

При недостатке информации ИИ обязан запрашивать уточнение,
а не придумывать новые сущности или экраны.

13. Цель стандарта

Цель стандарта:

масштабируемость интерфейса при росте системы;

единообразие пользовательского опыта;

отражение жизненного цикла бизнес-объектов;

корректную генерацию UI с помощью ИИ;

снижение когнитивной нагрузки пользователей.
