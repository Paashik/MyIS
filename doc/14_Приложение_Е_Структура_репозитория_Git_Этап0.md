# Приложение Е
## Рекомендуемая структура репозитория Git (Этап 0)

### 1. Назначение
Документ описывает рекомендуемую структуру репозитория исходного кода для реализации Этапа 0 (Каркас системы) и последующего развития проекта.

Цель — обеспечить понятную организацию кода, удобство для разработки, сопровождения и расширения.

---

## 2. Общие принципы

- Один **монорепозиторий** для всей системы (backend + frontend + конфигурация).
- Чёткое разделение backend и frontend по каталогам.
- Общие артефакты (документация, скрипты, CI/CD) — в корне или в отдельных папках.
- Именование директорий и проектов — латиницей, без пробелов.

---

## 3. Верхнеуровневая структура

```text
repo-root/
  backend/
  frontend/
  docs/
  ops/
  .gitignore
  README.md
```

### 3.1 `backend/`
Содержит решение .NET (модульный монолит).

Пример структуры:

```text
backend/
  src/
    Core.Domain/
    Core.Application/
    Core.Infrastructure/
    Core.WebApi/
  tests/
    Core.Domain.Tests/
    Core.Application.Tests/
```

- `Core.Domain` — доменные модели, сущности, бизнес-правила;
- `Core.Application` — сервисы прикладной логики (use cases);
- `Core.Infrastructure` — реализация репозиториев, доступ к PostgreSQL, миграции;
- `Core.WebApi` — ASP.NET Core Web API (контроллеры, запуск приложения);
- `tests/` — проекты модульных/интеграционных тестов.

### 3.2 `frontend/`
Содержит React-приложение.

Пример структуры:

```text
frontend/
  src/
    app/
      layout/
      routing/
      store/
    modules/
      auth/
      shell/
      requests/   (будущее)
      ...
    shared/
      components/
      hooks/
      utils/
  public/
  package.json
```

- `app/layout` — реализация App Shell (Header, Sider, Content);
- `app/routing` — конфигурация маршрутов;
- `modules/auth` — экраны логина и логика аутентификации;
- `modules/shell` — общие компоненты оболочки;
- `modules/requests` — модуль заявок (Этап 1 и далее);
- `shared` — общие компоненты, хуки, утилиты.

### 3.3 `docs/`
Документация по проекту.

```text
docs/
  concept/
  stage0/
  stage1/
  ...
```

Здесь могут храниться Markdown-версии Концепции, ТЗ по этапам, схемы, диаграммы.

### 3.4 `ops/`
Инфраструктура и эксплуатация:

```text
ops/
  ci/
  cd/
  docker/
  k8s/
```

- `ci/` — конфигурации CI (например, GitHub Actions, GitLab CI);
- `cd/` — конфигурации деплоя;
- `docker/` — Dockerfile и docker-compose для разработки/тестов;
- `k8s/` — манифесты Kubernetes (если применимо).

---

## 4. Стандарты именования веток

Рекомендуется следующая схема:

- `main` — стабильная ветка, готовая к деплою;
- `develop` — основная ветка разработки (опционально, если команда >1-2 человек);
- `feature/<name>` — функциональные задачи;
- `bugfix/<name>` — исправления ошибок;
- `hotfix/<name>` — срочные исправления продакшена.

---

## 5. Файлы в корне репозитория

- `README.md` — краткое описание проекта, инструкции по запуску;
- `.gitignore` — исключения для Git (bin/obj, node_modules и т.п.);
- при необходимости — `LICENSE`, `CHANGELOG.md`.

---

## 6. Рекомендации по дальнейшему развитию

- При добавлении новых доменных модулей (Requests, Procurement и т.д.) — создавать отдельные папки/неймспейсы в `backend/src` и `frontend/src/modules`;
- Избегать дублирования кода, общий функционал выносить в `shared`;
- Документацию по API, архитектуре и т.п. складывать в `docs/`.

---

## 7. Итог
Предложенная структура репозитория обеспечивает ясное разделение ответственности, понятна для новых разработчиков и масштабируема по мере роста системы.
