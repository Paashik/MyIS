# Приложение В
## Каркас интерфейса (Этап 0)

### 1. Назначение документа
Документ описывает структуру пользовательского интерфейса, которая создаётся на Этапе 0 и используется всеми последующими модулями системы. Данный каркас (App Shell) является единым и неизменным фундаментом для UI.

---

## 2. Общая архитектура интерфейса
Интерфейс реализуется в виде **SPA (Single Page Application)** на **React + Ant Design**. Основу составляет App Shell, включающий:

- **Header (верхняя панель)** — глобальная навигация и управление пользователем;
- **Sider (боковое меню)** — структура доменов системы;
- **Content (рабочая область)** — отображение текущей функциональности;
- **Login Page** — отдельная страница без Shell.

---

## 3. Компоненты App Shell

### 3.1 Header
Содержит:
- название системы / логотип;
- ФИО текущего пользователя;
- меню профиля: «Профиль», «Выход»;
- индикатор роли (опционально);
- переключатель темы (светлая/тёмная) — опционально.

Требования:
- Header отображается всегда, кроме экрана логина;
- должен быть адаптивным при сужении окна.

---

### 3.2 Sider (боковое меню)
Sider содержит основные домены системы:
- Requests — Заявки;
- Customers — Заказы клиентов;
- Procurement — Закупка;
- Production — Производство;
- Warehouse — Склад;
- Engineering — Инженерный контур;
- Technology — Технологический контур.

Требования:
- меню сворачивается (collapse) до компактного вида;
- пункты меню отображаются на основе ролей текущего пользователя;
- структура меню конфигурируется централизованно.

---

### 3.3 Content (рабочая зона)
Основная зона отображения UI каждого модуля.
Требования:
- поддержка вложенной навигации (React Router);
- единая система заголовков страниц;
- единые стили форм, таблиц, кнопок.

---

### 3.4 Login Page
Отдельная страница, отключённая от App Shell.
Элементы:
- логин
- пароль
- кнопка входа
- сообщения об ошибках

Требования:
- читабельный минималистичный интерфейс;
- фокус на поле логина при загрузке;
- сообщения о неверном пароле без технических деталей.

---

## 4. Навигация и маршрутизация

### 4.1 Основные маршруты (уровень 1)
- `/login` — авторизация;
- `/` — домашняя страница (может быть Dashboard);
- `/requests`;
- `/customers`;
- `/procurement`;
- `/production`;
- `/warehouse`;
- `/engineering`;
- `/technology`.

### 4.2 Требования к навигации
- Без перезагрузки страницы (SPA);
- История браузера должна корректно работать (кнопки «Назад»/«Вперёд»);
- Защищённые маршруты — недоступны без авторизации.

---

## 5. UI-компоненты и стандарты

### 5.1 Используемые библиотеки
- React
- React Router
- Ant Design

### 5.2 Требования к визуальной консистентности
- единые цвета и отступы;
- таблицы в Ant Design Table;
- формы — через Form + Input / Select;
- карточки — через Card;
- диалоги — через Modal.

### 5.3 Поддержка темизации
На Этапе 0 — опционально, но архитектурно предусматривается возможность.

---

## 6. Ролевая модель интерфейса (упрощённая)

### 6.1 Роли
- ADMIN — полный доступ;
- USER — доступ только к разрешённым модулям.

### 6.2 Правила
- пункты меню скрываются, если у роли нет прав доступа;
- переход по URL без прав должен выдавать страницу 403.

---

## 7. Требования к производительности
- Shell должен отображаться не более чем за 1 секунду;
- переходы между страницами — менее 300 мс при типичной нагрузке.

---

## 8. Требования к масштабируемости интерфейса
- возможность добавления новых пунктов меню без изменений в коде компоновки Shell;
- возможность внедрения подмодулей внутри доменов.

---

## 9. Итог
Каркас интерфейса Этапа 0 определяет единый стиль, структуру навигации и принципы отображения данных. Все последующие этапы обязаны использовать данный UI-фрейм как основу своей реализации.
